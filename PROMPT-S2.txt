## 🎯 OBJETIVO PRINCIPAL

Implementar sistema de autenticação IRC completo:
- **PASS**: Validar senha de conexão
- **NICK**: Definir nickname único
- **USER**: Capturar dados do usuário
- **State Machine**: Controlar transições (INIT → AUTH → ID → REGISTERED)
- **RPL_WELCOME (001)**: Confirmar autenticação

**Resultado esperado**: Cliente conecta via irssi/nc, autentica com PASS → NICK → USER, recebe RPL_WELCOME, pronto para S3.

## 🎯 TAREFAS (S2-T1 até S2-T6)

Cada tarefa tem spec técnica completa em [S2-AUTHENTICATION-INSTRUCTIONS.md](.github/docs/S2-AUTHENTICATION-INSTRUCTIONS.md).

### S2-T1: Comando PASS
- [ ] Handler `_handlePASS()`
- [ ] Validar senha contra `Server::_password`
- [ ] Se correto: marcar autenticado (estado AUTH)
- [ ] Se errado: enviar erro 464 e desconectar
- [ ] Teste: `test/S2-T1-pass-validation.sh`

**RFC**: Nenhuma resposta se OK; erro 464 se wrong

### S2-T2: Comando NICK
- [ ] Handler `_handleNICK()`
- [ ] Validar: não vazio, sem espaços, não duplicado
- [ ] Se válido: setar nickname (estado ID ou REGISTERED)
- [ ] Se duplicado: enviar erro 433 (não desconectar)
- [ ] Teste: `test/S2-T2-nick-validation.sh`

**RFC**: Error 433 (ERR_NICKNAMEINUSE) se duplicado

### S2-T3: Comando USER
- [ ] Handler `_handleUSER()`
- [ ] Parse: `USER <username> <mode> <unused> :<realname>`
- [ ] Extrair: username, realname
- [ ] Se válido: setar user + realname
- [ ] Se inválido: enviar erro 461
- [ ] Teste: `test/S2-T3-user-validation.sh`

**RFC**: Error 461 (ERR_NEEDMOREPARAMS) se falta parâmetro

### S2-T4: State Machine
- [ ] Enum: `ClientState {INIT, AUTH, ID, REGISTERED}`
- [ ] Transições: INIT → AUTH (PASS) → ID (NICK) → REGISTERED (USER)
- [ ] Método helper: `canTransitionToREGISTERED()`
- [ ] Validar ordem: PASS obrigatório antes de NICK/USER
- [ ] Teste: `test/S2-T4-state-machine.sh`

**Fluxo**: 
```
Cliente conecta (INIT)
  → PASS OK (AUTH)
  → NICK OK (ID)
  → USER OK (REGISTERED) ← RPL_WELCOME aqui
```

### S2-T5: RPL_WELCOME (001)
- [ ] Método `_sendWelcome()` 
- [ ] Formato: `:server 001 <nick> :Welcome to ft_irc\r\n`
- [ ] Enviado quando estado = REGISTERED (NICK+USER+PASS OK)
- [ ] Enviado uma única vez por cliente
- [ ] Teste: `test/S2-T5-welcome.sh`

**RFC**: Código 001, formato exato, com mensagem boas-vindas

### S2-T6: Testes com irssi + nc
- [ ] nc: conectar, PASS, NICK, USER, receber 001
- [ ] irssi: conectar, autenticar, receber welcome
- [ ] Múltiplos clientes simultâneos
- [ ] Wrong password rejeita
- [ ] Duplicate nick rejeita
- [ ] Teste: `test/S2-T6-irssi-validation.sh`

**Validação**: irssi mostra welcome message, nick aparece em `/w`

## 🔄 FLUXO RFC ESPERADO

### Cenário 1: Autenticação OK (Alice)
```
Alice → "PASS secret\r\n" → Server: valida, AUTH state
Alice → "NICK alice\r\n" → Server: ✓ ID state, responde ":server NICK alice"
Alice → "USER alice 0 * :Alice\r\n" → Server: ✓ REGISTERED state
                                      → Envia ":server 001 alice :Welcome to ft_irc"
```

### Cenário 2: Senha Errada (Bob)
```
Bob → "PASS wrong\r\n" → Server: ✗ erro 464, fecha conexão
```

### Cenário 3: Nick Duplicado (Charlie)
```
Charlie → "PASS secret\r\n" → Server: AUTH
Charlie → "NICK alice\r\n" → Server: ✗ erro 433 (alice já existe), não desconecta
Charlie → "NICK charlie\r\n" → Server: ✓ ID state
```

**Detalhes completos em**: [S2-AUTHENTICATION-INSTRUCTIONS.md - Exemplos de Fluxo IRC](.github/docs/S2-AUTHENTICATION-INSTRUCTIONS.md#-exemplos-de-fluxo-irc)
